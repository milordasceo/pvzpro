# Правила для Cursor AI

**Дата создания:** 31 декабря 2025 г. 20:32:56

## Общие правила общения

1. **Язык общения**
   - Отвечай всегда только на русском языке

2. **Стиль работы**
   - Чем меньше кода, тем лучше
   - Используй бэст практики
   - Будь последовательным

3. **Объяснения**
   - Объясняй максимально просто, пошагово, как для начинающего
   - Ассистент берёт на себя реализацию и предлагает подтверждать шаги

4. **Детальность инструкций**
   - Пользователь предпочитает очень детальные, пошаговые инструкции для начинающих

## Работа с проектом

### Инициализация

- При инициализации проекта используйте корневую директорию как сам проект
- Не создавайте отдельную поддиректорию для проекта
- Все команды должны выполняться из корневой директории проекта

### Документация

- При создании любого проекта обязательно создавать папку `docs` с тремя файлами:
  1. `README.md` - описание приложения и подробное описание каждого модуля и функции
  2. `context.md` - файл контекста для быстрой ориентации ИИ ассистента в проекте при новой чат-сессии из-за ограничений размера контекста
  3. `tasks.md` - список задач проекта
- Эти файлы должны актуализироваться после выполнения каждой задачи

### Синхронизация документации

- При изменении любого документа или важного файла проекта ОБЯЗАТЕЛЬНО проверить и синхронизировать ВСЕ связанные файлы
- Алгоритм:
  1. Идентифицировать все файлы, которые могут содержать похожую информацию (концепция, README, техстек, архитектура)
  2. Проверить каждый файл на предмет устаревшей информации
  3. Обновить все связанные упоминания технологий, версий, дат
  4. Убедиться в консистентности между всеми документами проекта
- Особенно важно для версий технологий, дат, названий компонентов, архитектурных решений

### Обновление контекста между сессиями

- Если пользователь пишет: "Подготовься к началу новой сессии" (или аналогично), ассистент обязан обновить все файлы в `docs/` (CURSOR_CONTEXT.md, PROJECT_OVERVIEW.md, TASKS.md, DESIGN_SYSTEM.md, PROJECT_MODULES.md) с актуальной датой/временем и состоянием проекта, чтобы новая сессия начиналась с актуальным контекстом

### Работа с задачами

- После каждой выполненной задачи необходимо:
  1. Отдать результат выполнения пользователю для тестирования
  2. Дождаться утверждения пользователя
  3. Только после утверждения переходить к следующей задаче
  4. Нельзя выполнять несколько задач подряд без промежуточного утверждения

- Документацию проекта можно обновлять только ПОСЛЕ получения утверждения от пользователя о правильно выполненной задаче
- Нельзя обновлять документацию до тестирования и утверждения пользователем

## Технологический стек

### Проверка версий

- Перед выбором любой технологии, зависимости или фреймворка ИИ проверяет текущую дату через терминал и определяет последнюю стабильную версию на эту дату

- **Обязательный алгоритм работы с технологическими стеками:**
  1. Сначала проверить текущую дату через терминал
  2. Выполнить веб-поиск актуальных версий каждой технологии на эту дату (например "React Native latest version August 2025")
  3. Проверить официальные релизы на GitHub/npm
  4. Убедиться что указываются СТАБИЛЬНЫЕ версии, а не альфа/бета
  5. Всегда уточнять "на [текущая дата]" в документации
  6. Никогда не полагаться только на общие знания о версиях без актуальной проверки

- При работе с проектом необходимо всегда проверять версии зависимостей на совместимость
- Для этого можно использовать веб-поиск или Context7 для получения актуальной информации о совместимых версиях пакетов

### Конкретные технологии

- Проект использует React Native с Expo на последней доступной версии (по состоянию на дату проекта)
- Всегда указывать актуальную текущую дату при указании технического стека

### UI-first подход

- Пользователь использует UI-first подход: сначала верстаем все экраны (дизайн и layout), затем натягиваем бэкенд
- Приоритет на создание полного UI/UX интерфейса перед интеграцией с API и базой данных

### UI библиотеки

- Использовать только React Native Paper для UI компонентов на всех экранах
- Запрещается использовать другие UI библиотеки или произвольные кастомные UI решения
- После внедрения React Native Paper 5.x с Material Design 3 количество кода стилей значительно сократилось (~170 строк StyleSheet уже убрано)
- Primary buttons в проекте должны быть стилизованы одинаково с кнопкой "Start Shift" и использоваться последовательно во всех основных действиях

### Типографика и дизайн

- UI типографика и размещение элементов должны соответствовать стандартным гайдлайнам дизайна проекта
- Использовать экран 'Overview' в качестве примера

## Работа с датами и временем

- Перед записью даты и времени в любые файлы документации или код необходимо всегда сначала узнать актуальную дату и время через терминальную команду (например, `date` или `Get-Date` в PowerShell)
- Это обеспечивает точность временных меток в документации проекта

## Терминал и команды

- Пользователь предпочитает, чтобы каждая терминальная команда выполнялась отдельно
- Терминал должен закрываться после завершения каждой команды перед запуском следующей
- Если терминальная команда не удалась, ассистент должен повторить попытку или попробовать другой подход вместо остановки

## Git и версионирование

- Пользователь требует, чтобы ассистент обновлял файл `git-status.html` при выполнении задач контроля версий или версионирования

## Прочая информация

- У пользователя нет Mac и нужны решения для создания проектов без Mac

- Пользователь подчеркнул, что не нужно добавлять функционал, который не был запрошен

## Архитектурные решения

- Документация должна отражать переход на Paper компоненты и новый подход к стилизации
- После выполнения каждой задачи необходимо обновлять все соответствующие файлы документации и контекста проекта

---

**Важно:** Все правила должны применяться последовательно и учитываться при каждой работе с проектом.
